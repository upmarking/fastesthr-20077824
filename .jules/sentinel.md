## 2024-03-24 - HTML Injection in PDF Templates
**Vulnerability:** XSS/HTML Injection vulnerability in `src/lib/pdf-generator.ts` where unescaped user inputs (e.g., candidate names, custom template variables) were directly substituted into HTML strings before being rendered to PDF and persisted.
**Learning:** The PDF generator natively executes `<script>` tags found in templates for features like CTC calculations before capturing `.innerHTML`. Using robust sanitizers like DOMPurify on the full template would strip these required scripts and break the feature.
**Prevention:** Rather than sanitizing the entire compiled template block or pulling in heavy dependencies, escape individual variable values (e.g., `&`, `<`, `>`, `"`, `'`) specifically during variable substitution. This ensures user input is safe while preserving the template's intentional functional elements.
## 2024-05-24 - Backreference Injection in String.prototype.replace()
**Vulnerability:** A Regex backreference injection vulnerability in `String.prototype.replace(regex, value)` was found in template variable substitution functions (`OfferLetterRenderer.tsx`, `pdf-generator.ts`, `send-offer-letter/index.ts`). If a user inputs `$` characters (like `$&` or `$1`) into their name or variables, JavaScript evaluates them as capture group references instead of literal strings, allowing data corruption and potential injection vectors in the HTML output.
**Learning:** `String.prototype.replace()` interprets special `$` patterns in the replacement string. When replacing variables with user-controlled input, passing the string directly exposes the application to unintended template mutations or XSS if sanitization relies on exact string matching.
**Prevention:** Always use a replacer callback function `() => value` instead of passing the string directly when substituting untrusted variables. Callbacks treat the return value as a literal string, completely neutralizing `$` backreference patterns.
